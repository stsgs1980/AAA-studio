## 1. Введение

**3A Studio** — это интегрированная среда разработки (IDE) для создания, управления и мониторинга мультиагентных AI-систем с визуальным drag-and-drop редактором. Платформа объединяет в единую живую систему то, что ранее требовало трёх отдельных репозиториев, 110+ навыков и ручной синхронизации между ними.

### Ключевые возможности

- **Визуальный редактор потоков** — 18 типов узлов в 6 категориях, drag-and-drop, branch-aware исполнение
- **Prompt Studio** — 5 модулей для инженерии промптов: Write, Formulas, Frameworks, Compare, Intent
- **База знаний** — коллекции документов с TF-IDF поисковым движком
- **Skills и Standards** — управление навыками и стандартами с кросс-валидацией
- **Самокоррекция** — цикл Generate → Judge → Revise с LLM-оценкой
- **Multi-provider LLM** — поддержка Z.ai, OpenAI, Anthropic, OpenRouter + кастомные провайдеры
- **Аналитика** — дашборд с KPI, тепловой картой, спарклайнами, автообновлением
- **Иерархия агентов** — визуальное дерево parent/child с drag-and-drop перепривязкой
- **i18n** — английский и русский языки (~310 строк перевода на каждый)
- **Экспорт** — Skills (MD/ZIP), Standards (ESLint JSON/YAML), Prompts (JSON/YAML/MD), Wiki (GitHub MD)

### Демо-доступ

По умолчанию создана учётная запись **admin / admin**. Данные сидируются при первом запуске через `prisma/seed.ts` (10 агентов, 4 потока, навыки, стандарты и т.д.).

## 2. Архитектура проекта

3A Studio построен как **монорепозиторий** на базе bun/pnpm workspaces с четырьмя внутренними пакетами и основным Next.js-приложением. Архитектура следует принципу **feature-based** организации: каждый модуль (agents, knowledge, skills и т.д.) выделен в отдельную директорию `src/features/<name>/` со своими компонентами, стилями и локальной логикой.

### Высокоуровневая схема

```
┌──────────────────────────────────────────────────────────────────┐
│                        Next.js App Router                        │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌───────────────────┐  │
│  │  Public  │  │  Auth   │  │Platform │  │    API Routes     │  │
│  │  Pages   │  │  Pages  │  │ Pages   │  │  (32 endpoints)   │  │
│  └────┬─────┘  └────┬────┘  └────┬────┘  └────────┬──────────┘  │
│       │              │            │                 │             │
│  ┌────▼──────────────▼────────────▼─────────────────▼──────────┐ │
│  │                    Middleware (JWT Auth)                      │ │
│  └──────────────────────────┬──────────────────────────────────┘ │
│                             │                                    │
│  ┌──────────────────────────▼──────────────────────────────────┐ │
│  │                   Feature Modules (17)                       │ │
│  │  landing · auth · dashboard · flow-editor · agents          │ │
│  │  agent-creator · prompt-studio · knowledge · skills         │ │

│  └──────────────────────────┬──────────────────────────────────┘ │
│                             │                                    │
│  ┌──────────────────────────▼──────────────────────────────────┐ │
│  │                 Monorepo Packages                            │ │
│  │  @stsgs/shared · @stsgs/ui · @stsgs/prompting              │ │

└──────────────────────────────────────────────────────────────────┘
```

### Принципы проектирования

- **Feature-first** — каждая фича инкапсулирована в `src/features/<name>/`
- **API-first** — все операции идут через REST API с Zod-валидацией
- **Cross-ref validation** — удаление сущности проверяет все ссылки (409 Conflict)
- **Edge middleware** — JWT-проверка на уровне Edge для защиты маршрутов
- **Circuit Breaker** — автоматическая защита от каскадных сбоев LLM-провайдеров

## 5. Установка и запуск

### Предварительные требования

- **Node.js** >= 18
- **pnpm** >= 8
- **Git**

### Шаги установки

```bash
# 1. Клонирование репозитория
git clone <repo-url> 3a-studio
cd 3a-studio

# 2. Установка зависимостей (монорепозиторий)
pnpm install

# 3. Копирование переменных окружения
cp .env.example .env
# Отредактируйте .env: укажите DATABASE_URL, AUTH_SECRET, ENCRYPTION_KEY

# 4. Миграция и сидирование БД
npx prisma migrate dev --name init
npx prisma db seed

# 5. Запуск в режиме разработки
pnpm dev
```

### Переменные окружения

| Переменная | Обязательная | Описание |
|-----------|:---:|---------|
| `DATABASE_URL` | Да | URL подключения к БД (SQLite: `file:./dev.db`, PostgreSQL: `postgresql://...`) |
| `AUTH_SECRET` | Да | Секретный ключ для JWT-подписи (минимум 32 символа) |
| `ENCRYPTION_KEY` | Да | Ключ шифрования API-ключей (AES-256-GCM, 32 байта hex) |
| `ADMIN_EMAIL` | Нет | Email админа для сидирования (по умолчанию: `admin@3a.studio`) |
| `ADMIN_PASSWORD` | Нет | Пароль админа для сидирования (по умолчанию: `admin`) |
| `ZAI_API_KEY` | Нет | Z.ai API-ключ для production (авто-инъекция при отсутствии ключа в БД) |

### Доступные скрипты

```bash
pnpm dev          # Запуск dev-сервера с Turbopack
pnpm build        # Production-сборка
pnpm start        # Запуск production-сервера
pnpm lint         # ESLint-проверка
pnpm test         # Запуск Vitest
pnpm db:migrate   # Миграция Prisma
pnpm db:seed      # Сидирование БД
pnpm db:studio    # Prisma Studio (визуальный редактор БД)
```

## 7. Аутентификация и безопасность

### JWT-аутентификация

Система аутентификации построена на библиотеке `jose` с алгоритмом HS256. При входе пользователь получает JWT-токен, который сохраняется в httpOnly cookie с сроком действия 7 дней. Edge middleware проверяет токен на каждом защищённом маршруте.

**Поток аутентификации:**

1. Пользователь отправляет POST `/api/auth/login` с email и паролем
2. Сервер валидирует учётные данные, подписывает JWT (HS256) с payload `{ userId, email, role }`
3. Токен устанавливается в httpOnly cookie `session` с `maxAge: 7d`, `sameSite: lax`
4. Edge middleware проверяет cookie на каждом запросе к защищённым маршрутам
5. При выходе (POST `/api/auth/logout`) cookie очищается

### Шифрование API-ключей

API-ключи LLM-провайдеров шифруются алгоритмом **AES-256-GCM** перед сохранением в базу данных. Ключ шифрования берётся из переменной окружения `ENCRYPTION_KEY`. В UI ключи отображаются маскированно (показываются только последние 4 символа).

### Защита маршрутов

Edge middleware (`src/middleware.ts`) проверяет JWT для всех маршрутов, начинающихся с `/dashboard`, `/editor`, `/agents`, `/prompt-studio`, `/knowledge`, `/skills-page`, `/standards`, `/pipelines`, `/settings`, `/wiki`, `/templates`, `/tasks`, `/workflows`, `/hierarchy`, `/agent-creator`, `/quality-analyzer`, `/self-correction`, `/audit`. Публичные маршруты (`/`, `/login`, `/signup`, `/verify`, `/forgot-password`, `/reset-password`) доступны без аутентификации.

## 8. Модели данных (Prisma)

3A Studio использует 37 Prisma-моделей, организованных в логические группы. В режиме разработки используется SQLite, в продакшене — PostgreSQL (Neon).

### Агенты

**Agent** — центральная сущность системы. Содержит имя, роль, группу ролей, статус (draft/active/archived), модель, температуру, maxTokens, systemPrompt, формулу, инструменты, привязанные навыки и стандарты. Имеет self-referential иерархию (parentId) и связь «близнец» (twinId).

**AgentExecution** — журнал выполнения агента. Содержит статус, входные/выходные данные (JSON), длительность, количество использованных токенов.

**Task** — задача, привязанная к агенту. Содержит заголовок, описание, статус, приоритет.

### Потоки и визуальный редактор

**Flow** — визуальное определение потока. Хранит узлы и рёбра в JSON, имеет версию и метаданные.

**FlowVersion** — история версий потока. Каждое обновление создаёт новую версию.

**PipelineExecution** — запись выполнения потока. Хранит статус и результат (JSON).

### Рабочие процессы

**Workflow** — определение многошагового рабочего процесса с триггером и тегами.

**PipelineStep** — отдельный шаг рабочего процесса с агентом, действием, схемами входа/выхода, условием и политикой повторов.

**WorkflowExecution** — экземпляр выполнения рабочего процесса.

**StepExecution** — запись выполнения отдельного шага с входными/выходными данными и результатом мышления агента.

**AgentMessage** — межагентные сообщения в рамках выполнения шага с верификационным статусом.

### Промпт-система

**PromptHistory** — история оценки промптов. Хранит промпт, намерение, уверенность, формулу, средний балл, вердикт, количество шагов.

**PromptTemplate** — шаблон промпта с переменными, фреймворком и тегами.

**PromptVersion** — версия промпта агента с changelog и флагом активности.

**PromptRegistryEntry** — запись в реестре промптов с версией и статусом.

### База знаний

**KnowledgeCollection** — контейнер коллекции документов с именем, описанием и тегами.

**KnowledgeDocument** — отдельный документ в коллекции с содержимым, типом файла и тегами.

## 12. База знаний (Knowledge Base)

Модуль базы знаний позволяет создавать коллекции документов и выполнять по ним семантический поиск.

### Структура

- **KnowledgeCollection** — именованный контейнер с описанием и тегами
- **KnowledgeDocument** — документ внутри коллекции с содержимым, типом файла и тегами

### TF-IDF поисковый движок

Встроенный поисковый движок на базе алгоритма **TF-IDF** (Term Frequency — Inverse Document Frequency) не требует внешней векторной БД. Реализован полностью на TypeScript.

**Как работает:**

1. Документы разбиваются на токены (words)
2. Вычисляется TF (частота терма в документе) для каждого слова
3. Вычисляется IDF (обратная частота документа) — редкость слова во всей коллекции
4. TF × IDF даёт вес каждого слова в контексте документа
5. Поисковый запрос векторизуется тем же способом
6. Косинусное сходство между вектором запроса и векторами документов определяет релевантность

## 16. Самокоррекция

Модуль самокоррекции реализует цикл **Generate → Judge → Revise** для итеративного улучшения вывода агента.

### Как работает

1. **Generate** — агент генерирует начальный ответ на основе входных данных
2. **Judge** — LLM-судья оценивает ответ по шкале 0-10 с вердиктом:
   - `approved` — ответ принимается, цикл завершается
   - `needs_revision` — ответ требует доработки
   - `rejected` — ответ отклонён
3. **Revise** — агент получает оценку судьи и обратную связь, генерирует улучшенный ответ
4. Цикл повторяется, пока вердикт не станет `approved` или не будет достигнут лимит попыток

### Параметры

- **Max retries** — максимальное количество циклов ревизии (0-5, по умолчанию 3)
- **Judge score threshold** — порог баллов для автоматического одобрения

| POST | `/api/self-correction` | Запуск сессии самокоррекции |

### Настройка провайдера

Выбор активного провайдера и модели осуществляется в **Settings** (`/settings`). API-ключи шифруются AES-256-GCM перед сохранением и отображаются маскированно в UI.

### Уровень устойчивости (Resilience)

Система имеет встроенный уровень устойчивости к сбоям LLM-провайдеров:

**Exponential Backoff Retry:**
- Начальная задержка: 1 секунда
- Множитель: 2x
- Максимальное количество попыток: 3
- Jitter: случайная добавка для предотвращения thundering herd

**Circuit Breaker:**
- Состояния: CLOSED → OPEN → HALF_OPEN → CLOSED
- Порог открытия: 5 последовательных сбоев
- Таймаут HALF_OPEN: 60 секунд
- Автоматический переход в CLOSED после успешного запроса

**Fallback Manager:**
- При недоступности основного провайдера автоматически переключается на резервный
- Конфигурируемый список резервных провайдеров

## 20. Кастомный ESLint-плагин

Пакет `@eslint-plugin-3a` содержит 4 кастомных правила для поддержания качества кода:

### max-lines (150)

Ограничивает количество строк в файле. Предотвращает создание «божественных» файлов и стимулирует декомпозицию. Значение по умолчанию — 150 строк, может быть переопределено в конфигурации.

### max-use-state (3)

Ограничивает количество `useState`-хуков в одном компоненте. Если компоненту нужно больше 3 стейт-переменных, это сигнал к рефакторингу — выносу логики в кастомный хук или использованию Zustand-стора.

### no-cross-layer

Запрещает неправильные импорты между слоями архитектуры. Например, компонент из `features/` не может напрямую импортировать что-то из другого фич-модуля — только через shared-пакеты. Предотвращает появление неявных зависимостей между фичами.

### no-unicode-escapes

Запрещает использование unicode-escape последовательностей (например, `\u00b2`, `\u2728`). Все спецсимволы должны использоваться в литеральном виде или через соответствующие React/HTML-механизмы.

## 22. Сущностные связи

```
Standard ←─(standardIds)──→ Skill ←─(skills)──→ Agent → Flow → PipelineExecution
Standard ←────(standards)─── Agent
Agent → AgentExecution
Agent → Agent (parent/child self-relation via parentId)
Agent → Agent (twin relation via twinId)
Flow → FlowVersion
KnowledgeCollection → KnowledgeDocument
Workflow → PipelineStep → StepExecution → AgentMessage
Task → Agent
SelfCorrectionSession → Agent
PromptTemplate → (используется в Prompt Studio)
```

### Правила удаления

При удалении любой сущности система проверяет все перекрёстные ссылки. Если сущность используется другой сущностью, возвращается **409 Conflict** с указанием ссылающихся объектов. Каскадное удаление применяется только к прямым дочерним сущностям (например, удаление коллекции удаляет все её документы).

## 25. Глоссарий

| Термин | Определение |
|--------|------------|
| **3A** | Artificial. Agentic. Architecture. — название и философия проекта |
| **Агент (Agent)** | Автономная AI-сущность с ролью, промптом и инструментами |
| **Поток (Flow)** | Визуальный граф из узлов и рёбер, определяющий последовательность действий |

| **Формула (Formula)** | Структурированный шаблон промпта (RTF, RISE, CO-STAR и др.) |
| **Фреймворк (Framework)** | Пошаговый билдер промпта с интерактивным UI |
| **Пайплайн (Pipeline)** | Выполнение потока с трекингом прогресса и результатов |
| **Рабочий процесс (Workflow)** | Многошаговый конвейер с последовательным выполнением |
| **Самокоррекция (Self-Correction)** | Цикл Generate → Judge → Revise для итеративного улучшения |
| **Circuit Breaker** | Паттерн устойчивости, предотвращающий каскадные сбои |
| **Fallback** | Автоматическое переключение на резервный LLM-провайдер |
| **TF-IDF** | Term Frequency — Inverse Document Frequency, алгоритм поиска |
| **Entity Picker** | UI-компонент для привязки Skills/Standards к сущностям |
| **Branch-aware** | Способность выполнения потока учитывать ветвление (Router/Condition) |
| **Human-in-the-Loop** | Узел, требующий ручного подтверждения перед продолжением |
| **Topo-sort** | Топологическая сортировка узлов перед выполнением потока |
